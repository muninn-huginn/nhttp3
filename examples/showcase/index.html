How HTTP/3 Works

Connection Setup

StepHTTP/1.1HTTP/2HTTP/3
TCP Handshake1.5 RTT1.5 RTTNot needed (UDP)
TLS Handshake2 RTT (TLS 1.2)1-2 RTTCombined with transport
Total3.5 RTT2.5-3.5 RTT1 RTT
At 100ms RTT350ms250-350ms100ms

Head-of-Line Blocking

ScenarioHTTP/1.1HTTP/2HTTP/3
10 parallel requests10 TCP connections1 TCP, multiplexed1 QUIC, independent streams
1 packet lost1 request stallsALL requests stallOnly affected stream stalls
Streaming responsesBlocks othersBlocks others (TCP)Independent delivery

Network Resilience

EventHTTP/1.1 + HTTP/2HTTP/3
WiFi → CellularConnection drops, reconnectConnection migrates seamlessly
Packet lossTCP retransmit, window collapsePer-stream retransmit only
High latencyMultiple RTTs to start1 RTT to start

When HTTP/3 Helps Most

Use CaseImprovement
Mobile clients (200ms RTT)Saves 500ms per connection
Cross-continent API calls (100ms RTT)Saves 250ms per connection
LLM token streamingNo HOL blocking on concurrent streams
Lossy networks (WiFi, cellular)Only affected stream retransmits
Localhost / same datacenterTCP is slightly faster (no TLS overhead)
HTTP/3 is not universally faster. It wins on latency and resilience, not throughput. On localhost with 0ms RTT, TCP is faster because QUIC's TLS handshake has overhead that can't be amortized.
